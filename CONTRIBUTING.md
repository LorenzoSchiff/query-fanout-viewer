# Contributing

Thanks for considering a contribution.

## What is most useful

- provider payload fixes
- parser hardening
- UI readability improvements
- documentation improvements
- reproducible bug reports
- security-minded reviews

## Before opening a pull request

1. Explain the problem clearly
2. Describe the provider and context
3. Include a minimal reproducible example when possible
4. Keep changes scoped
5. Avoid unrelated refactors

## Pull request style

- small and focused
- descriptive title
- explain what changed
- explain why it changed
- include screenshots for UI changes

## Bug reports

A strong bug report includes:
- provider name
- browser
- normal or private browsing
- expected behavior
- actual behavior
- console error, if any
- sanitized payload snippet, when possible
